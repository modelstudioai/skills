# MCP协议、插件与数据连接对比

在百炼平台构建智能应用时，开发者常面临如何高效接入外部工具、业务 API 与企业数据的架构选型问题。`[[MCP协议]]`、`[[插件]]` 与 `[[数据连接]]` 均旨在拓展大模型的能力边界，但三者在设计初衷、数据交互范式、生命周期管理及计费逻辑上存在显著差异。本文旨在为开发者提供结构化对比与技术选型参考，明确各方案的适用边界与最佳实践，帮助您根据业务需求快速匹配最优架构。

## 核心维度对比

| 对比维度 | [[MCP协议]] | [[插件]] | [[数据连接]] |
|:---|:---|:---|:---|
| **核心定位** | 标准化模型上下文交互协议，实现跨平台工具/数据源无缝对接与动态路由 | 预置或自定义 HTTP API 工具集，弥补模型在计算、搜索、执行等方面的短板 | 企业数据源统一接入与管理层，支持结构化/非结构化数据的检索与实时查询 |
| **输入格式** | JSON-RPC / HTTP 请求体，支持动态参数解析、多步编排与结构化注入 | HTTP `GET`/`POST`，支持 `application/json` 或 `x-www-form-urlencoded` 编码 | 自然语言标签过滤 (`tags`)、SQL 语句、文件/表格直接上传或流式挂载 |
| **输出格式** | 结构化数据块（直接拼接入对话上下文，可能显著增加 Input Token） | 模型依据出参结构过滤重组的 API 响应（支持 Object 嵌套，严禁空子属性） | 向量化切片、表格行数据、数据库查询结果集（以 RAG 上下文或工具返回值形式透传） |
| **支持模型与调用** | 不可直接通过 `[[千问API]]` 调用；必须依托 `[[智能体应用]]` 或 `[[工作流应用]]` 编排触发 | 支持通义千问系列（Turbo/Plus/Max/VL 等）；可通过 Assistant API 或应用直调路由 | 绑定至 `[[智能体应用]]`/`[[工作流应用]]` 作为知识库或实时查询源，不依赖特定模型版本 |
| **API 端点/协议** | 标准格式：`https://.../mcps/<service>/mcp`；底层全面升级支持 `streamableHttp` | 平台内置分发网关，依赖 `Tool ID` 标识与标准 RESTful 规范 | 控制台可视化接入；底层依赖 EventBridge/DTS 实现网络连通与逻辑同步 |
| **鉴权与安全机制** | 容器级隔离；Header 注入 `Bearer <API_KEY>`；支持 `[[AI网关]]` 代理鉴权 | Header/Query 鉴权（Basic/Bearer/AppCode）；支持服务级静态 Token 与用户级动态传入 | 阿里云 RAM 权限管控；VPC 私网互通/公网白名单；语雀/数据库原生凭证托管 |
| **计费方式** | **自定义脚本**：基础模式按运行时长 `0.000156 元/秒`；极速模式含基础驻留费 `0.000036 元/秒`+调用费<br>**官方服务**：免部署费，按量调用 | 按 API 实际调用次数计费（官方部分免费，第三方/自定义依云市场定价） | **平台托管**：限时免费/超额度按量；**流处理**：依底层数据库读写、DTS 同步及存储组件计费 |
| **典型场景** | 复杂多步工具路由、云端异步任务执行、跨生态 REST API 标准化封装、IDE 本地工具链集成 | 实时联网搜索、数学计算、代码解释执行、轻量级第三方 SaaS 快速接入 | 企业知识库构建（PDF/Office 解析向量化）、MySQL/PG 业务库实时状态查询、OSS 海量文件流式读取 |

## 各方案适用场景建议

### 🟦 选择 `[[MCP协议]]` 的场景
- **需要跨客户端/生态的统一集成**：希望同一套工具配置能同时赋能 `[[智能体应用]]`、`[[工作流应用]]` 以及本地 IDE（如 Cursor、Cherry Studio）。
- **复杂逻辑与异步任务**：业务涉及多步参数依赖、云端脚本执行（Node.js/Python），或需通过 `[[函数计算]]`/`[[AI网关]]` 代理现有微服务。
- **技术提示**：自定义服务运行在无状态云端环境，**无固定公网 IP**，访问 RDS 等需配置白名单/VPC；旧版 SSE 已弃用，生产环境务必使用新版 `streamableHttp` 端点。若大模型未触发工具，请优先检查 System Prompt 指令明确度或切换至强推理模型（如 qwen-max）。

### 🟩 选择 `[[插件]]` 的场景
- **开箱即用与快速验证**：需要调用平台预置能力（如 `calculator`、`code_interpreter`、`quark_search`），无需编写底层通信逻辑即可实现模型自动调度。
- **标准同步 HTTP 调用**：业务 API 为典型的请求-响应模式，入参可通过大模型识别或 `biz_params` 显式透传，且对出参结构有明确过滤需求。
- **技术提示**：新版控制台已逐步将自定义插件迁移至 `[[MCP协议]]` 管理。修改插件 URL/Header 或参数后**必须重新发布**；`code_interpreter` 等沙箱环境严格隔离外网。上线前建议针对不同模型版本的触发策略进行边界测试。

### 🟨 选择 `[[数据连接]]` 的场景
- **知识库与 RAG 构建**：核心诉求是管理企业非结构化文档（PDF/Word/PPT），利用文档切片与向量化技术实现高精度语义检索。
- **业务数据库直连查询**：需实时获取 MySQL、PostgreSQL 或 PolarDB-X 中的交易/状态数据。**注意：仅通过 DMS 导入的数据源支持直接执行 SQL**。
- **技术提示**：表格结构（列名/类型）创建后**不可修改**，后续导入文件需严格对齐；PostgreSQL 需开启 `wal_level=logical` 并放行 `100.64.0.0/16` 网段；OSS 挂载需避开归档/冷归档存储类型。平台托管存储额度与记录展示周期（90天）需结合业务增长提前规划。

## 开发者技术选型决策树

```text
业务诉求是什么？
├─ 需标准化集成外部 API/脚本，且希望跨端复用、支持异步/多步编排？
│  └─ ✅ 采用 [[MCP协议]]（注意评估冷启动延迟与上下文 Token 成本）
├─ 仅需轻量级同步工具（搜索、计算、代码执行、简单 SaaS 调用）？
│  └─ ✅ 采用 [[插件]]（关注 Tool ID 规范与鉴权动态注入）
└─ 核心是访问/解析企业自有数据（文档、表格、关系型数据库）？
   └─ ✅ 采用 [[数据连接]]（区分平台托管解析 vs 流处理 SQL 直查）
```

> **💡 最佳实践建议**：在大型复杂应用中，三者常组合使用。例如：通过 `[[数据连接]]` 沉淀企业知识库实现基座检索，结合 `[[MCP协议]]` 托管业务核心 API 与自动化脚本，并将高频轻量操作交由 `[[插件]]` 快速路由。建议在架构设计初期明确数据流向与计费归属，避免上下文膨胀与 Token 成本失控。
